# Homework 2 设计演进文档

## 一、提示功能实现

### 1.1 实现方式
我将提示功能严格分为两层：

**Sudoku 层（纯逻辑）**：在 `@sudoku/sudoku.js` 中实现了两个核心纯函数：
- `getCandidates(sudoku, x, y)`：计算指定格子的所有可能候选数
- `getNextHint(sudoku)`：使用"唯一候选数"和"隐式唯一数"两种策略寻找下一步可填的格子

**Game 层（会话管理）**：在 `hints.js` store 中实现：
- 管理提示次数的消耗和重置
- 管理当前提示的UI状态
- 提供统一的接口给UI组件调用

### 1.2 提示功能的归属
**提示功能既属于 Sudoku，也属于 Game**，两者职责清晰：

- **候选数计算和逻辑推断属于 Sudoku**：这是数独本身的数学属性，不依赖于任何游戏状态。这些逻辑被实现为纯函数，可以独立测试和复用。
- **提示次数消耗和 UI 交互属于 Game**：这是游戏会话层面的控制，与用户的游戏进度和设置相关。

这种分离设计符合单一职责原则，提高了代码的可维护性和可测试性。

## 二、探索模式的实现

### 2.1 实现思路
我选择了 **"Game 进入一种新的状态"** 的设计思路。

- 创建了独立的 `explore.js` store 来管理探索状态
- 进入探索时，保存当前棋盘快照和历史记录长度作为"探索起点"
- 探索过程中，正常使用现有的 Undo/Redo 机制
- 提交时，直接将当前状态合并入主历史
- 放弃时，精确回滚到探索起点快照

### 2.2 主局面与探索局面的关系
**共享对象，逻辑隔离**：

- 我们不进行深拷贝整个游戏状态，而是共享同一个 `userGrid` store
- 探索过程中的所有操作都直接作用于主棋盘和主历史记录
- 提交时不需要特殊操作，因为探索过程已经在主线上
- 放弃时，通过记录的历史长度精确撤销探索期间的所有操作

这种设计避免了深拷贝带来的性能问题和引用污染问题，同时保持了代码的简单性。

### 2.3 冲突检测与失败路径记忆
- 每次用户操作后自动检查棋盘冲突
- 实现了失败路径记忆功能：记录已经探索失败的棋盘状态
- 当用户再次进入已经失败的路径时，自动提示冲突

## 三、History 结构的演进

### 3.1 变化
- 从无到有：Homework 1 中 Undo/Redo 只是 UI 占位符，Homework 2 中实现了完整的线性栈 history
- 功能增强：支持精确回滚到任意历史点，为探索模式提供了基础

### 3.2 设计
仍然使用线性栈结构，**未引入树状分支**。

探索模式被视为"在主线上进行的一段可回滚的操作"。这种设计的优点是：
- 保持了历史记录机制的简单性
- 原有 Undo/Redo 功能完全不受影响
- 实现成本低，不易出错

## 四、实现难点与权衡

我选择了线性栈+精确回滚的设计，而不是树状历史结构，主要基于以下考虑：
1.  **复杂度权衡**：树状结构会大大增加历史记录管理的复杂度，而线性栈足够满足本次作业的需求
2.  **Svelte特性**：Svelte的响应式系统对简单的线性数据结构支持更好
3.  **用户体验**：对于普通用户来说，线性的Undo/Redo更加直观
4.  **作业要求**：题目明确说明"不要求多层嵌套探索"和"DAG合并"

这种设计在满足所有作业要求的同时，保持了代码的简单性和可维护性。

## 五、Homework 1 设计的局限

1. **缺少完整的 History 机制**：Homework 1 中 Undo/Redo 只是空实现，无法支持任何回滚操作
2. **Hint 功能过于简单**：原来的 `applyHint` 只是直接调用求解器填答案，没有任何逻辑推断
3. **缺少状态隔离机制**：没有考虑过"临时状态"和"永久状态"的分离，无法支持探索模式
4. **领域逻辑与UI逻辑耦合**：部分逻辑分散在UI组件中，没有提取到纯函数中

## 六、如果重做 Homework 1

1. **一开始就引入 History 机制**：将 Undo/Redo 作为核心功能设计，而不是后期添加
2. **将 Sudoku 逻辑纯净化**：更早地将候选数计算、冲突检测、提示推断等逻辑提取到独立的纯函数中
3. **引入状态机设计**：明确游戏的各种状态（正常、探索、暂停、结束）和状态转换规则
4. **增加事件系统**：让不同模块之间通过事件通信，而不是直接调用，降低耦合度
